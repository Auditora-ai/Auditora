## BUG-022: TypeScript error — `name: null` incompatible with Prisma `Organization.name: String`

**Severity:** High  
**Component:** `apps/saas/app/api/organization/profile/route.ts` (line 66)  
**Found by:** QA Agent #05  
**Date:** 2026-04-02  
**Status:** 🔴 OPEN

---

### Steps to Reproduce

1. TypeScript compile: `pnpm --filter saas exec tsc --noEmit`
2. Observe error on line 66 of `profile/route.ts`

### Expected Behavior

No TypeScript errors. The `companyName` field should only pass a valid `string` (never `null`) to Prisma's `organization.update()`.

### Actual Behavior

```
app/api/organization/profile/route.ts(66,3): error TS2322:
  Type '{ ... name?: string | null | undefined; }' is not assignable to Prisma OrganizationUpdateInput
  Type 'null' is not assignable to type 'string | StringFieldUpdateOperationsInput | undefined'
```

### Root Cause

In the Prisma schema, `Organization.name` is `String` (non-nullable). The fix for BUG-011 added:

```typescript
companyName: z.string().nullable().optional()
```

And then:

```typescript
...(companyName !== undefined ? { name: companyName } : {})
```

This allows `companyName = null` to be passed as `name: null`, which violates Prisma's non-null constraint.

### Code Reference

File: `apps/saas/app/api/organization/profile/route.ts`  
Lines: 26, 64–66  

```typescript
// Zod schema allows null:
companyName: z.string().nullable().optional(),

// Spread into Prisma update allows null to reach name field:
...(companyName !== undefined ? { name: companyName } : {}),
// ↑ if companyName === null, passes { name: null } → TypeScript error
```

Also in Prisma schema:
```
model Organization {
  name String  // NOT nullable
```

### Suggested Fix

Option A (preferred): Change Zod schema to NOT allow null for companyName, only undefined:
```typescript
companyName: z.string().min(1).optional(),
```

Option B: Add null guard before spreading:
```typescript
...(companyName ? { name: companyName } : {}),
```

Option A is better as it aligns with the schema constraint and validates the input more strictly.

### Regression Risk

Medium — runtime behavior only breaks if the API caller sends `companyName: null`. The TypeScript error prevents this from being caught at compile time. The `pnpm build` still passes because Next.js ignores TS errors at build time, meaning this is a silent type safety gap.
